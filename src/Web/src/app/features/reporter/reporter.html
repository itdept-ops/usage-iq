Self-service — get your own reporter token

You can mint and revoke ingest keys for your own usage — no admin needed. Each key is tied to your account, so anything it reports shows up under {{ auth.session()?.email }} and your machine on the Fleet page.

  1. Generate a key below and copy it (it's shown only once).
  2. Pick a reporter for the machine where your AI coding agent runs — the desktop agent (a friendly tray app, step 2) is the easiest, or the headless console reporter for servers (step 3).
  3. Point it at this server with your key — it parses the logs locally and pushes only token counts & metadata, never prompts or responses.

2Desktop agent (Windows) easiest

The Usage IQ desktop agent is a system-tray app that wraps the reporter in a friendly GUI — no terminal babysitting. Paste this server's URL and your key once in its Settings screen and it watches your logs in the background, with a live log of every sync and a run-at-sign-in toggle so it comes back after a reboot.

settingsOne-screen Settings — URL, key, machine label, interval.
articleLive logs — watch each pass scan, price, and post in real time.
restart_altRun at sign-in — starts minimized to the tray after a reboot.
Run it from source (needs the .NET 9 SDK)
{{ agentRunCommand() }}
…or publish a single, self-contained tray exe to hand out (no .NET install on the target)
{{ agentPublishCommand() }}
info The published UsageIq.Agent.exe needs no key on the command line — generate one above, then paste it into the agent's Settings screen on first launch.
3Console reporter (headless / cron)

For servers and anything without a desktop, the reporter is a small .NET 9 console app in the repo (src/Reporter). Build a release binary, then point it at this server with the key from step 1. By default it watches the logs continuously; pass --once for a single pass (e.g. from cron).

git clone https://github.com/itdept-ops/usage-iq && cd usage-iq dotnet build src/Reporter -c Release
Then run the built binary:
{{ runCommand() }}
…or straight from source:

Keep the console reporter running

Run the watcher under your OS service manager so it survives reboots, or schedule --once on a timer. Config can also come from REPORTER_* env vars or an appsettings.json next to the executable. (On a desktop, the agent in step 2 handles all of this for you — this section is for headless hosts.)

terminal Linux (systemd)
[Service] Environment=REPORTER_URL={{ serverUrl() }} Environment=REPORTER_KEY=uiq_… ExecStart=/opt/usage-iq/usage-iq-reporter Restart=always
schedule Windows (Task Scheduler)
schtasks /create /tn "UsageIQ Reporter" /sc onlogon ^ /tr "usage-iq-reporter.exe --url {{ serverUrl() }} --key uiq_…"
event_repeat cron (hourly --once)
0 * * * * usage-iq-reporter \ --url {{ serverUrl() }} --key uiq_… --once

On Windows, Run-UsageIqReporter.ps1 in the repo is a one-file "build + run the loop" launcher — the headless counterpart to the desktop agent. The reporter merges the redundant log lines each turn emits before sending, so a backfill is a few hundred requests, not thousands — full details, service setup, and troubleshooting are in the reporter guide.

Where your usage shows up

Whichever reporter you run, the usage it pushes is attributed to its machine and to the account that owns the key. You'll see both — ranked by spend, with the machines and users cross-linked — on the Fleet page.
